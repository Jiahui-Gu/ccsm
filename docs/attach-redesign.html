1 · UX 模型 —— 三种状态

右侧内容区就这三种样子。左边的 sidebar 是固定的。

State 0

刚打开 ccsm

session A
session B
session C
— 空 —
右侧全黑。还没人 attach 过,什么都没准备。
State 1

冷启动中

session A
session B
session C
(底层旧 shell,被盖住)
准备中…
遮罩盖住右侧。后台拉 snapshot、建 xterm、写入、滚到底,好了揭开。遮罩是每个 shell wrapper 自己的,不是右侧 host 共用的 —— 切到别的 session 不会把这个遮罩带过去。
State 2

已 attach 切换

session A

$ npm test
PASS ./foo.test.ts
▍
右侧是 z-stack。点谁谁拉到顶,各自保留上次离开的位置,无遮罩、无冷启动。

状态怎么流转

State 0  ──点第一个 session──▶  State 1  ──准备完──▶  State 2
                                                       │
                                                       ├──点已看过──▶ 留在 State 2(秒切)
                                                       │
                                                       └──点没看过──▶ State 1(再来一次冷启动)

2 · 两条规则

用户行为只分两种,响应也只有两种。

规则 1
第一次点 = 冷启动

右侧盖一层 "准备中" 的遮罩。背后拉 snapshot、建 xterm、写入、滚到底,全部做完才揭开。用户不会看到半成品。

规则 2
再点已看过的 = 秒切

右侧 z-stack 里把这个 session 翻到顶,viewport 还停在上次离开的位置。无遮罩,无加载,无重建。

3 · 背后做什么(high-level)

不展开 IPC 和状态机,只说大方向。

  • 没点过的 session 完全不准备 —— 不订 PTY,不建 xterm,什么资源都不占。
  • 点过一次之后 留住不放 —— xterm 实例和 PTY 订阅都不 dispose,所以再切回来是秒级。
  • 关 ccsm 才回收 —— 退出时一次性 dispose 所有留下来的 shell。
  • 右侧是 z-stack —— 看过的 shell 都叠在那里,点哪个哪个拉到顶。
  • 看过的 session 一直留着,关 ccsm 才一次性回收。

为什么留住不放

服务端 (@xterm/headless + SerializeAddon) 一直在 main 进程里保留每个 session 的镜像,renderer 这边理论上随时 dispose 再重建都行。但本设计选择留住,因为切换体感差别明显:z-stack swap 0ms,重建一次大约 ~50ms(还要等 snapshot 往返)。这是有意识的权衡。如果以后内存真成问题再加回收逻辑,目前不做。

5 · 其他操作的行为

sidebar 上的三个操作:删除、重做、复制。都不引入新的 UX 概念,只是套用前面两条规则。

操作 1
删除 (delete)

· sidebar 那行消失。
· 后台 dispose 这个 session 的 xterm,解订 PTY。
· 删的是当前顶层 → 顶层移走,z-stack 下一层自动露出(上次看的那个)。
· 删的是 z-stack 里非顶层 → 用户看不到任何变化。
· 删的是从没看过的 session → 啥都不发生,本来就没建。
· 删的是正在冷启动的 session → cancel,遮罩消失,顶层退回下一层。

操作 2
重做 (reload)

· 原地重置,不 dispose:同一个 xterm 实例、同一个 wrapper、同一组 listener。只是 term.reset() + 清 buffer + 重拉 snapshot + 写入 + 滚到底。
· 重做的是顶层 → 该 wrapper 自己的遮罩亮起来(蓝色"重做中…"),走完揭开。
· 重做的是非顶层 → 不显示遮罩(用户也看不到),后台静默走完。万一用户中途切过来,看到的就是 reload 当前进行到哪一步的样子 —— 接受这个 corner case,不专门处理。
· reload 走的是同一条路径,顶层 / 非顶层的区别只在"要不要显示遮罩"。

操作 3
复制 (copy session)

· 调用方拿到新 sid(已有产品行为,不改)。
· 复制的是当前顶层 session → 新 sid 立刻打开,走第一次点击冷启动(顶上来),原 session 下沉。
· 复制的是非顶层 → 新 sid 只在 sidebar 出现,不自动抢顶层,等用户主动点才冷启动。
· 复制的是从没看过的 session → 新 sid 在 sidebar 出现,不自动打开。

6 · Sidebar 高亮

每个 session 行可能有一个蓝色圆点,语义就一条:蓝点 = z-stack 顶层,也就是右侧当前看得见的那个 session。只表示 "现在显示的就是它"。

规则
activeSid = zStack[0]

· activeSid = z-stack 顶层 sid。蓝色圆点跟着 activeSid 走。
· State 0(右侧全黑):没有任何行有圆点。
· State 1(冷启动中):用户点的那行有圆点(因为它即将上顶)。
· State 2(秒切):z-stack 顶层 sid 那行有圆点。
· 删除顶层 session:顶层 pop,下一层 sid 变成新顶层 → 圆点自动跳到 sidebar 那行。
· 非顶层 reload / 非顶层 copy:顶层不变,圆点不动。
· 实现一句话:activeSid = zStack[0],sidebar 订阅这个值。

7 · 并发与取消

长跑 operation(冷启动 / reload)期间用户可能操作其他东西。下面是统一的处理方式 —— 两个 stop 信号,一个 shouldStop 检查点,没有 case-by-case 的 if/else。

两条规则

规则 1

Shell 还存在吗?
长跑 operation 在每个 await 之后检查 shell.disposed。disposed=true 就退出。delete 操作不直接 cancel operation,它只调 disposeShell,operation 自己会发现并退出。

规则 2

这个 operation 还是 "最新的" 吗?
同一 shell 同种 operation 新的覆盖旧的。每个 operation 启动时把自己写到 shell.currentOp。旧的在 await 之后看到 currentOp !== self 就退出。

实现模式

async function reload(sid) {
  const shell = getShell(sid);
  if (!shell) return;
  const self = (shell.currentOp = {});  // 用对象身份做 token

  if (shell.isTop) shell.showMask();    // 非顶层不显示遮罩

  shell.pausePtySubscription();         if (shouldStop(shell, self)) return;
  await drain();                        if (shouldStop(shell, self)) return;
  term.reset();                         if (shouldStop(shell, self)) return;
  const snap = await fetchSnapshot();   if (shouldStop(shell, self)) return;
  term.write(snap);                     if (shouldStop(shell, self)) return;
  term.scrollToBottom();
  shell.resumePtySubscription();

  if (shell.currentOp === self) {
    shell.currentOp = null;
    shell.hideMask();
  }
  // 没到这里就 return = 被新 op 顶替或 shell 已 dispose。
  // 遮罩 / 订阅 / currentOp 都不在这里清理 ——
  // 下一个 op 接手时会自己 showMask / setCurrentOp,
  // disposeShell 兜底统一释放。
}

function shouldStop(shell, self) {
  return shell.disposed || shell.currentOp !== self;
}

整个 codebase 只有 shouldStop 一个判断函数。每个 await 后都检查一次,幂等。被打断时直接 return,不做任何清理 —— 清理是下一个 op 或 disposeShell 的事。不要用 try/finally 在 cancel 路径上 hideMask,disposed 的 shell 引用可能已经无效,容易崩。

关键不变量

  • 遮罩是每个 shell wrapper 自己的 —— 不是右侧 host 共用一个。被打断的 op 直接 return,不清理遮罩;下一个 op (showMask 或不 show)接手,disposeShell 兜底。
  • 切 session 不动正在跑的 operation —— 只改 activeSid。operation 完成时一次性判断 "我跑完了,activeSid 还是我吗?是 → showShell;不是 → 不抢顶层"。这是完成时的一次判断,不是 cancel 逻辑。
  • delete = 调 disposeShell —— operation 自己看到 disposed 退出,delete 不需要知道有哪些 operation 在跑。

term.reset() 跟 PTY chunk 怎么不打架

每个 shell 一个简单的串行 op 序列(冷启动 / reload / dispose 都是 op)。平时 PTY chunk 直接写进 xterm,跟 op 无关。reload 这个 op 一启动就先 pause PTY 订阅(chunk 暂时不送进 xterm,在 main 端缓冲),然后 reset → fetchSnapshot → write snapshot → resume 订阅(main 把暂存的 chunk 一次性补回来,接上 snapshot 之后的内容)。这样 reset 跟 chunk 不会交错,xterm 看到的永远是 "snapshot + 之后顺序到达的 chunk"。

冷启动时 chunk 比 xterm 早到怎么办

顺序写死成:先建 xterm + open + 写 snapshot,再 subscribe PTY。snapshot 是 main 端某个时间点的镜像,subscribe 启动后 main 会从那个点之后的 chunk 开始往下发(main 一直在缓冲,没丢)。这样根本不存在 "chunk 来了 xterm 还没好" 的窗口,也不需要 renderer 端的 pending 数组。

为什么不用 queue 处理用户操作

上面说的 "op 序列" 是 shell 内部串行(reload 之间互相 supersede,跟 PTY chunk 的同步用 pause/resume 解决)。但用户连点 reload 时不排队 —— 用户后一下就是想覆盖前一下,FIFO 反而不对。新 reload 写入 currentOp,老的 shouldStop 拦下退出。两条规则 + shouldStop,够了。

corner case 自然推论

场景结果为什么
reload A 中途 delete A reload 自动退出 shell.disposed=true,shouldStop 拦下
reload A 中途用户切走 reload 继续跑完,但不抢顶层 shell 还在,operation 仍是 currentOp;跑完时检查 activeSid 决定是否 showShell
reload A 中途又 reload A 第一个 reload 退出,第二个跑 第二个写入 currentOp,第一个 shouldStop 拦下
冷启动 B 中途切回 A B 的冷启动继续跑;不抢顶层 同 reload 切走 case
冷启动 B 中途 delete B 冷启动自动退出,wrapper 不出现 shell.disposed=true
copy A 中途 delete A 不相干 copy 操作的是 newSid,A 的 delete 不影响 newSid

8 · 实现要点

给后续 worker 看的落地清单。不是完整方案,只是关键决策。

每个 shell 的结构

每个 visited session 对应一个 Shell 对象:一个 wrapper div、wrapper 里挂一个 xterm 实例、wrapper 里盖一个自己的遮罩 div、一个 PTY 订阅、currentOp、disposed 标志。遮罩跟着 wrapper 走,不在 host 层共享 —— 切 session 时遮罩状态留在原 wrapper 里,新顶层 wrapper 自己有自己的遮罩(默认隐藏)。

z-stack 怎么实现

右侧 host 里放 N 个 wrapper div,每个 wrapper 里挂一个 xterm。切换只动 display 和 z-index:目标 shell display:'',其余 display:'none'。xterm 实例从不在 wrapper 之间 reparent。

遮罩怎么做

每个 shell wrapper 里有一个 position:absolute; inset:0 的遮罩 div(不是 host 层共享)。冷启动开始时 display:'' 显示 "准备中…",建 xterm + 写 snapshot + 滚到底跑完后 display:'none' 揭开。秒切场景遮罩一直是 display:'none'。reload 顶层时同样亮自己 wrapper 的遮罩;reload 非顶层时不亮 —— 用户看不见,没意义。

PTY 订阅时机

第一次点 session 的顺序写死:建 xterm → term.open(wrapper) → fetchSnapshot → term.write(snapshot) → pty.open(sid) 开始订阅。snapshot 是 main 端某个时间点的镜像,subscribe 之后 main 把那个点之后缓冲的 chunk 顺序补回来,接上 snapshot。这样 renderer 这边永远不会出现 "chunk 来了 xterm 还没好" 的窗口,不需要在 renderer 端搞 pending buffer。订阅一旦挂上就一直挂着,直到关 ccsm 调 pty.close;脱离前台不取消订阅,后台 session 的输出持续写进 scrollback,秒切回来就在那里。

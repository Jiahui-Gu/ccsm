Attach 重新设计 —— 从零开始

如果今天从头写 ccsm 的 attach,会长成什么样。当前 19 步流程和上面的补丁全不看,只谈设计。
范围:pty attach + xterm mount + 侧边栏耦合 · 配套文档 docs/attach-timeline-review.html(历史)

1 · 设计原则

五条原则,后面所有决策都要回答它们。

P1
第一帧 = 最终帧。
用户看到画面的那一瞬间,画面就得是最终态:滚到对的行、字体已渲染好、尺寸已就位。当前问题是用户能看见 "空白 → 内容 → 跳一下" 的过程。新设计把所有准备工作藏在 display:'none' 后面,翻成 display:'' 就是结果。
P2
Terminal 是 per-sid 的长寿对象,不是 attach 的副产物。
xterm 实例、addon、字体预热、PTY 数据订阅 —— 都在我们第一次关心这个 sid 时一次性建好,之后一直留着。当前 "切会话就重新走一遍" 的代价直接消失。"Attach" 退化成 "把已经准备好的东西显示出来"。
P3
数据订阅与可见性无关。
PTY chunk 从进程启动到结束一直流进对应的 Terminal,不管它是否在屏幕上。当前为了等 attach 才订阅,中间有个 "buffering listener" 模式做缓冲 —— 这个中间层在新设计里直接没有,因为消费者永远在线。
P4
侧边栏走自己的数据通道。
侧边栏要的是 agent 状态、标题、徽标,这些早就由 session:state / session:title 单独推送。它和 PTY chunk 没关系,新设计继续保持这种解耦,绝不去解析 PTY 输出做侧边栏预览。
P5
能并行就并行,一次汇合,然后串行,最后 reveal。
独立的事情(拉 snapshot、做布局、算 viewport)同时跑,用一次 await Promise.all 等齐。真正有顺序要求的(写 snapshot → drain → 钉位)再串行。Reveal 是整个流程的最后一条指令。

2 · 总体架构

五个阶段,从左到右。前两个阶段在用户点会话之前就已经在跑。

Phase 0
Provision(lazy)
侧边栏里的 session 行进入可视区(IntersectionObserver)才 provision:在 offscreen 容器里建 xterm Terminal、加 addon、订阅 PTY 流。滚出可视区不 dispose,关闭 ccsm 才回收。归档分组里的 session 要展开 + 进入可视区才 provision —— 几百条归档不会一起吃内存。
Phase 1
持续 stream
已 provision 的 sid,PTY chunk 持续流进它的 Terminal,直到进程退出。侧边栏状态走另一条通道。谁在前台无所谓。
Phase 2
准备可见
用户点了某个 sid。并行做三件事:把 SessionShell reparent 到可见容器(仍 display:'none')、commit 布局拿到真实尺寸、算好 viewport 要停在哪。
Phase 3
Reveal
一条赋值:display:''。viewport、字体、尺寸、内容全是最终态。用户看到的就是这一帧。
Phase 4
Reveal 之后
把输入路由切到这个 sid、focus、装 ResizeObserver 基线。顺序无所谓,因为用户已经看到结果。

为什么这么分

当前 attach 把四件事 —— 建 Terminal、订阅数据、做布局、改可见性 —— 揉成一个大过程,所以又慢又容易出 bug。把它们拆开做,重活在用户点之前就跑完了。等用户点会话,字节其实已经流进 Terminal 几秒,所谓的 "attach" 就只剩 "把这个 Terminal 显示出来" 而已。

3 · Phase 2 内部的并行流

用户点会话后到 reveal 之前发生的事。仍 display:'none',所以做多少步用户都看不到。

Lane A · 数据
A1. 若 shell 尚未写过 snapshot,调 pty.snapshotAt(sid, sinceSeq?);否则跳过(已 live)。
A2. writeAsync(snapshot) 写入 Terminal scrollback。Drain。
A3. Drain 截至当前的 in-flight live chunk(一次显式 await term.write('', cb) barrier —— 全流程唯一使用处)。
Lane B · DOM
B1. 把 SessionShell 从 offscreen pool reparent 到可见 <TerminalSlot/>,仍 display:'none'。
B2. await rAF —— 让布局 commit。断言 clientWidth/Height > 0(否则进入错误路径 F3)。
B3. fit.fit() → 按真实尺寸算出最终 cols/rows。
Lane C · Viewport 计划
C1. 读 shell 的 lastViewportY(warm)或 null(cold)。
C2. 决定目标:warm → 恢复 Y;cold+resume → 贴底;cold+空 → 贴底(no-op)。
C3. 借 shell 的 scrollback 行数预算;此时不修改 terminal。
▼
await Promise.all([ A, B, C ]) 单点 barrier —— 三条 lane 全部就绪
▼
串行尾部 · 仍 display:'none'
S1. 若 fit.cols ≠ pty.cols:pty.resize(sid, cols, rows)。引发的 SIGWINCH 回显落在 live 流(已指向该 Terminal);不重新拉 snapshot —— parser 会追上。
S2. 一次 await term.write('', cb) drain —— 同时吸收 A3 的 live tail和 S1 的 SIGWINCH 回显。
S3. 应用 lane C 的 viewport 计划:term.scrollToLine(targetY) 或 term.scrollToBottom()。一次。精确。
▼
slot.style.display = '' 用户看到的第一帧 —— 已是最终态
▼
Phase 4 · reveal 之后 · 无序
P1. 在 store 翻转 activeSid → 输入路由开始把击键送到该 sid 的 pty.input。
P2. term.focus()。
P3. 安装 ResizeObserver 基线(吃掉 display 翻转引起的第一个伪 tick,与当前实现同理)。

依赖说明

  • A 和 B、C 互不依赖。A 可以在 B reparent 之前就写完 —— 写入进一个还没上屏的 Terminal,完全没问题。
  • B2(rAF)是唯一一处要等 DOM 的地方。便宜、可预测,通常一帧就够,<16ms。
  • C 是纯计算,单独成一条 lane 是为了让 warm/cold 的决策作为 S3 的明确输入,而不是塞在 S3 里一堆 if/else。

4 · 各场景下的第一帧保证

从用户角度只有两种情况:第一次点开有历史的会话,或者切回之前看过的会话。下面是各自看到什么、为什么不会出问题。

场景 A —— cold + resume第一次 attach 有历史的会话

第一帧:完整的 xterm 画布、尺寸最终、scrollback 已渲染到 snapshot 最后一行、滚动条贴底、光标停在 CLI 离开的位置。不会出现空帧、向上漂、迟到的 snap。
  • 怎么做到的:snapshot 写入(A1–A2)和 live tail drain(A3、S2)都在 reveal 之前完成。Viewport 位置在 C 算好,在 S3 一次性 scroll 到位。Reveal 排在 S3 后面。
  • 为什么不会 race:没有 fire-and-forget 的写。S2 是唯一可能有 pending 写的位置,这里我们 await 它。
不变量:display:'' 那一刻,term.buffer.baseY === lastWrittenBaseY 且 ydisp === baseY。前者来自 S2(无 pending 写入),后者来自 S3(恰好一次钉位)。

场景 B —— warm 切换之前 attach 过,再切回来

第一帧:用户离开时的 viewport 位置,包括停在 scrollback 中间。滚动条对齐。脱离期间新到的 chunk 在 viewport 下方,不会自动滚到。
  • 怎么做到的:SessionShell 一直没 dispose(P2)。Lane A 完全是 no-op —— chunk 一直在流。Lane C 读上次离开时记的 lastViewportY。S3 调一次 scrollToLine(targetY)。
  • 关于 reparent:reparent(B1)不会动 xterm 内部状态。仍然走 rAF(B2),防止换宿主后宽度变了。
不变量:同一个 sid,deactivate 前和 reactivate 后 renderer 状态一致,只有 baseY 可能因为 live chunk 往前涨过 —— 这是对的,新数据应在 buffer 里,只是不应拽动 viewport。

5 · IPC 契约提案

主进程和 renderer 之间的接口。比现在更少、更明确。

// ─── main → renderer 广播 ───────────────────────────────────────────────

// 单一通道,per-sid 一条流。从进程启动 emit 到 dispose。
// renderer 让对应的 Terminal 当唯一消费者;主进程不根据 "有没有人在看"
// 来 gate emit。正因如此,buffering-mode 那一套缓冲层就没必要了 —— 
// 消费者永远在。
channel 'pty:chunk': { sid: string; seq: number; data: Uint8Array }

// 生命周期。和 chunk 分开,消费者不用解析 chunk 去判断进程是否退出。
channel 'pty:exit': { sid: string; code: number | null; signal: string | null }

// 侧边栏自己的数据通道,和现状一致,这里列出来求完整。
channel 'session:state': { sid: string; state: SessionState }
channel 'session:title': { sid: string; title: string }

// ─── renderer → main RPC ────────────────────────────────────────────────

// 打开 SessionShell。幂等。何时调:
//   • 普通 session 行进入侧边栏可视区(IntersectionObserver 触发)
//   • 归档分组被展开 且 行进入可视区
// 不调:行滚出可视区(保留 shell)、关闭归档分组(保留 shell)。
// 退出时 close。返回 renderer 应视作 live 起点的 seq 游标。
rpc pty.open(sid): Promise<{ pid: number; cols: number; rows: number; seq: number }>

// 一次性历史 snapshot。返回的 snapshot 覆盖到 `seq`(不含)之前的全部
// 内容。renderer 先写 snapshot,再播放 seq >= returned.seq 的 chunk。
// 替代 'buffering listener' 那层间接。
rpc pty.snapshotAt(sid, opts?: { sinceSeq?: number }): Promise<{
  snapshot: Uint8Array;
  seq: number;     // `snapshot` 覆盖的上界(不含)
  cols: number;
  rows: number;
}>

// Resize 是 hint;若 cols/rows 和当前一致,主进程返回但不 emit SIGWINCH。
// 所以每次 fit() 后无脑调一下都安全。
rpc pty.resize(sid, cols, rows): Promise<{ applied: boolean }>

// 输入是 fire-and-forget,per-sid 有序。
channel 'pty:input': { sid: string; data: string }   // renderer → main,单向

// 丢掉 SessionShell。主进程可能让 PTY 继续活着(给其他窗口或后台),
// 但不再向本窗口 emit 该 sid 的 chunk。
rpc pty.close(sid): Promise<void>

// ─── 故意不要的东西 ─────────────────────────────────────────────────────
// • 没有 pty.attach / pty.detach。订阅就藏在 pty.open 里。
// • 没有 getBufferSnapshot/attach 拆分。换成 pty.snapshotAt,
//   任何时候调都行,返回自洽的 {snapshot, seq}。
// • 没有侧边栏预览通道。侧边栏只读 agent 状态。

为什么是这些形状

  • 叫 pty.open 不叫 pty.attach —— "open" 才符合事实:renderer 承诺一直消费到 pty.close。窗口生命周期里不会有反复 attach/detach,切 sid 完全不打扰主进程。
  • chunk 自带 seq —— snapshot 和 live 拼接天然对得上(seq < snapshotSeq 的丢掉,其余写)。不靠时间戳,不用问 "我有没有漏接"。
  • pty.snapshotAt 和 open 分开 —— renderer 想懒拉就懒拉(真要渲染历史时再调),想立刻拉也行,IPC 一样。不再绑死 "一 attach 就 snapshot"。
  • resize.applied 有返回值 —— renderer 不用猜 SIGWINCH 有没有发出来,主进程说了算。

6 · 每个 SessionShell 的 renderer 状态

用 discriminated union 表达。状态本身不容易出错,出错的都是状态转移,所以把它们写明。

type SessionShell = {
  sid: string;
  term: Terminal;            // xterm 实例,在 provision 时创建
  container: HTMLElement;   // 默认 offscreen;activate 时 reparent 到 slot
  lastViewportY: number | null; // 每次 deactivate 时记录;首次之前为 null
  status: ShellStatus;
};

type ShellStatus =
  | { kind: 'provisioning' }                                   // term 已创建,等待 pty.open
  | { kind: 'streaming'; seq: number }                         // pty.open 完成,chunk 流动,尚未写过 snapshot
  | { kind: 'hydrating'; snapshotSeq: number }                 // snapshot 已拉,正在写入
  | { kind: 'live'; baseSeq: number }                          // snapshot 已 drain,live chunk 直接应用
  | { kind: 'exited'; code: number | null }                    // 收到 pty.onExit;terminal 转为只读
  | { kind: 'error'; reason: ShellError; recoverable: boolean };

type ShellError =
  | 'open-failed'             // pty.open 被拒(claude 二进制缺失、spawn EPERM 等)
  | 'snapshot-failed'         // pty.snapshotAt 重试后仍失败
  | 'host-zero-size'           // rAF 已 settle,slot 仍为 0×0 —— 布局未给空间
  | 'stream-disconnected';    // chunk 通道静默,但未收到 exit 事件

// 可见性在 shell 之外 —— 正交轴:
type SlotState =
  | { kind: 'empty' }
  | { kind: 'mounting'; sid: string }       // 已 reparent,display:'none',跑 Phase 2 串行尾部
  | { kind: 'visible'; sid: string };       // reveal 之后

把 ShellStatus 和 SlotState 两轴叉乘,所有可能的情况都在表里。当前 attach 路径的大多数 bug 在这个模型里是非法态(比如已经可见但还没 live),新设计让这种态写不出来:mounting → visible 的切换和 display:'' 是同一个调用点,而这个点要求 shell.status.kind === 'live',编译器/类型系统就拦住了。

7 · 失败路径

每种错误都有名字、用户看到什么、怎么恢复。

失败用户看到什么恢复
F1 · open-failed Slot 保持空;xterm 画布位置换成内联错误卡:"无法为该会话启动 Claude。[重试] [查看日志]"。侧边栏行带小错误标点。 重试按钮再调一次 pty.open;第三次失败时弹出 claude-missing 指引(沿用现有 UI)。
F2 · snapshot-failed 如果 shell 从未写过 snapshot:同 F1 的错误卡。如果 shell 已经 live、这次是补一次 snapshot(罕见,比如 chunk 丢了):UI 不变,继续用当前 buffer,弹一个非阻塞 toast 提醒。 250ms 后自动重试一次;第二次失败升级成错误卡。
F3 · host-zero-size Slot 停在 mounting,显示一个淡 spinner(通常 <200ms)。超过 2s:错误卡 "终端区域没有空间 —— 试着调整窗口大小。" 每次 slot 的 ResizeObserver tick 都重排 rAF;一拿到非零尺寸,从 B3 继续走 Phase 2。
F4 · stream-disconnected 终端顶部横幅:"实时更新已暂停 —— 重连中…"。已有 scrollback 仍可读;输入禁用。 再调 pty.open 重新订阅;若 seq 跳跃,调 pty.snapshotAt(sid, {sinceSeq}) 做增量 snapshot 并应用。
F5 · exited 终端只读;光标变灰;侧边栏 agent 状态变 exited。底部 chip 显示退出码,带重启按钮。 重启 spawn 一个新 sid(复用 SessionShell 容器,装一个新的 Terminal);已有会话不会静默复活。
F6 · shell-not-ready 边缘情况 —— 用户刚加完 session 立刻点,Phase 0 的 provision 还没跑完。Slot 进入 mounting 显示淡 spinner(典型 <500ms),provision 完成立刻接着走 Phase 2。 不用用户操作。如果 provision 本身失败,降级为 F1。

8 · 与当前 19 步路径的对比

不是 diff,是结构对比。两栏,一句话一条。

当前

  • 19 个串行步骤,数据、DOM、布局、顺序全混在一起。
  • Terminal 生命周期和 attach 绑死:cold 和 warm 走两条代码路径。
  • 有个 buffering-listener 模式专门缓冲 attach 之前的 chunk。
  • 调了两次 scrollToBottom(L479 + L551),因为第一次太早。
  • fit 在 attach 之后才做,SIGWINCH 回显追在第一次钉位后面(QP-3)。
  • live-tail 写入是 fire-and-forget(QP-1)—— 最严重的 bug。
  • term.open 在过期或零尺寸布局里跑(QP-2),CanvasAddon atlas 当着用户面预热(QP-4)。

重设计

  • 5 个阶段,只有 Phase 2 内部有顺序,而且就是 "三 lane 并行 + 一次汇合"。
  • Terminal 只建一次、关 ccsm 才 dispose。切 sid 完全不动 xterm。
  • 没有 buffering 模式,因为 chunk 永远流进一个已存在的 Terminal。
  • 只有一次钉位:S3,drain 完之后、reveal 之前。
  • Fit 排在任何 resize 协商之前(B3 → S1),SIGWINCH 回显被 S2 的 drain 顺手吃掉。
  • 所有写入都 await,S2 的 drain 是结构性的,不是补丁。
  • term.open 在 rAF + 非零尺寸断言之后才跑,reveal 是最后一步,atlas 预热永远在用户看不到的地方。

新架构里压根不会发生的 race

  • QP-1(live tail vs 钉位):没有 fire-and-forget tail,S2 显式 drain,S2 和 S3 之间不写任何东西。
  • QP-2(stale-dim open):term.open 只在容器有真实尺寸时跑(P0 offscreen 容器有固定尺寸,B2 rAF 保护可见路径)。
  • QP-3(fit-after-attach SIGWINCH 回显):fit 在 pty.resize 之前算好,resize 的回响被 S2 吃掉。
  • QP-4(用户能看见 atlas 预热):atlas 在 Phase 0 就预热完了 —— 那时容器在 offscreen,用户根本看不见。
  • StrictMode 双 mount(QP-5):provision 以 sid 为 key 幂等,effect 重跑就是 no-op。

可以删掉的代码

  • src/terminal/usePtyAttach.warm.ts —— 19 步过程,被一个只翻 SlotState 的小 hook useVisibleShell(sid) 取代。
  • xtermWarmRegistry.ts::ensureAndShowEntry / applySnapshot / pinViewportToBottom —— 折成 shellRegistry,只暴露 provision/close/getSnapshot。
  • "buffering" listener 模式 —— 数据面解耦之后没人需要它了。
  • pty:attach 和 pty:detach IPC 通道 —— 由 pty.open / pty.close 取代,per-window 的流是隐式的。
  • snapshot 前的防御性 term.reset() —— shell registry 保证 terminal 要么是新的、要么已 live,reset 没人调。

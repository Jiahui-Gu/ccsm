Attach redesign — from scratch

If we wrote ccsm’s attach path today, what would it look like? Discard the 19-step incumbent and the patches on top of it. Design only.
scope: pty attach + xterm mount + sidebar coupling · companion to docs/attach-timeline-review.html (history)

1 · Design principles

Five things every later decision answers to.

P1
First frame = final frame.
Before display:'', the user perceives nothing. After it, the user must see a finished picture: scrolled to the right line, atlas warm, dims final. Step count between mount and reveal is irrelevant; the cut line is what matters.
P2
The Terminal is a long-lived per-sid object, not a per-attach artifact.
Allocation, term.open, addon load, atlas warm-up, and the live data subscription happen the first time we have any reason to care about a sid (e.g. it’s in the sidebar). “Attach” is reduced to making one of these visible. Switching sessions never disposes or re-opens xterm.
P3
Data subscription is independent of visibility.
Per-sid chunks flow into the per-sid Terminal continuously, from spawn/restore to dispose. There is no “buffering listener” mode that toggles around attach — the listener is always the Terminal itself. Visibility is a DOM concern only.
P4
Sidebar runs on its own data plane.
Sidebar state (agent state, title, badge, archived flag) comes from session:state / session:title — already independent of PTY chunks. Redesign keeps it that way; no last-line preview parsing of PTY output anywhere.
P5
Parallel where independent. One barrier. Then serial. Then reveal.
Independent work (snapshot fetch, layout commit, addon warm-up) runs concurrently in invisible space. A single await Promise.all meets them. The serial tail does only the bits with real ordering (write snapshot → drain live tail → pin viewport). Reveal is the last instruction.

2 · High-level architecture

Five phases. Read left to right.

Phase 0
Provision
When a sid enters the sidebar, allocate a renderer-side SessionShell: xterm Terminal in an offscreen container, addons loaded, live stream subscription opened. No user-visible side effect.
Phase 1
Stream-live
PTY chunks land in the per-sid Terminal continuously. Sidebar receives session:state independently. Both run forever, regardless of which sid is active.
Phase 2
Prepare-visible
User picks a sid. Parallel: reparent the SessionShell into the visible host (still display:'none'), commit layout, fit to real dims, restore last viewport (warm) or snap to bottom (cold).
Phase 3
Reveal
One assignment: display:''. Viewport, atlas, dims, content — all already final. This is the only frame the user sees.
Phase 4
Post-reveal
Wire input passthrough (it’s already wired at the Terminal layer; we only flip a routing flag), focus, mark active in store, install/refresh ResizeObserver baseline.

Why this shape

The current path conflates four orthogonal things — Terminal lifecycle, data subscription, layout, and visibility — into one mega-procedure called “attach.” Untangling them moves all the heavy lifting out of the click path. By the time the user clicks a session, the bytes have already been streaming for seconds; “attach” becomes “show.”

3 · Parallel flow inside Phase 2

What can run concurrently while still in display:'none', what must serialize, and where reveal sits.

Lane A · Data
A1. Request pty.snapshotAt(sid, sinceSeq?) if shell hasn’t written a snapshot yet, else skip (already live).
A2. Apply writeAsync(snapshot) into the Terminal’s scrollback. Drain.
A3. Drain in-flight live chunks up to now (single explicit await term.write('', cb) barrier — only place we use it).
Lane B · DOM
B1. Reparent SessionShell from offscreen pool into visible <TerminalSlot/>, still display:'none'.
B2. await rAF — let layout commit. Assert clientWidth/Height > 0 (else go to error path F3).
B3. fit.fit() → compute final cols/rows from real dims.
Lane C · Viewport plan
C1. Read shell’s lastViewportY (warm) or null (cold).
C2. Decide target: warm → restore Y; cold+resume → bottom; cold+empty → bottom (no-op).
C3. Precompute via shell’s scrollback line count; no terminal mutation yet.
▼
await Promise.all([ A, B, C ]) single barrier — three lanes fully settled
▼
Serial tail · still display:'none'
S1. If fit.cols ≠ pty.cols: pty.resize(sid, cols, rows). The resulting SIGWINCH echo lands in the live stream (already going to this Terminal); we don’t re-fetch a snapshot — the parser will catch up.
S2. One await term.write('', cb) drain — catches both the live tail from A3 and any SIGWINCH echo from S1.
S3. Apply viewport plan from C: term.scrollToLine(targetY) or term.scrollToBottom(). Once. Exact.
▼
slot.style.display = '' user’s first frame — already final
▼
Phase 4 · After reveal · order-free
P1. Flip activeSid in store → input routing now sends keystrokes for this sid to pty.input.
P2. term.focus().
P3. Install ResizeObserver baseline (eat the first spurious tick from display flip, same rationale as today).

Dependency notes

  • A is independent of B/C. A can complete before B reparents — the writes go into a Terminal that just isn’t on screen yet, which is fine (Phase 1 / P3).
  • B2 (rAF) is the only DOM-timed barrier. It is cheap and predictable: at most one frame, typically <16ms.
  • C is pure computation. It exists as its own lane to make the warm-vs-cold decision an explicit data input to S3 rather than a branching mess inside it.

4 · First-frame guarantees per scenario

The only two scenarios that exist from the user’s point of view. What they see and the invariant that makes it true.

Scenario A — cold + resumefirst attach of a session with history

First frame: full xterm canvas at final size, scrollback rendered to the last line of the resume snapshot, scrollbar at bottom, cursor where the CLI left it. No empty frame, no upward drift, no late snap.
  • How: snapshot write (A1–A2) and live-tail drain (A3, S2) complete before reveal. Viewport target is computed in C (bottom) and applied in S3 once. Reveal happens after S3.
  • Why no race: there is no fire-and-forget write. The single drain barrier in S2 is the only point where parser queue can be non-empty before reveal, and we wait on it explicitly.
Invariant: at the moment of display:'', term.buffer.baseY === lastWrittenBaseY and ydisp === baseY. Both follow from S2 (no pending writes) and S3 (exactly one pin).

Scenario B — warm switchpreviously attached, switching back

First frame: the exact viewport position the user left, including mid-scrollback positions. Scrollbar aligned. If new chunks arrived while detached, they are in the scrollback below the viewport (not auto-scrolled to).
  • How: the SessionShell was never disposed (P2). Lane A is a no-op — chunks have been streaming into this Terminal the whole time. Lane C reads the persisted lastViewportY (captured on each visibility-loss). S3 calls scrollToLine(targetY).
  • Reparent considerations: reparent (B1) does not reset xterm internal state. We still gate on rAF (B2) to be safe against width changes between hosts.
Invariant: the renderer state for sid X is identical immediately before deactivation and immediately after reactivation, save for baseY which may have advanced from live chunks (which is correct — they should be in the buffer, just not pulling the viewport).

5 · IPC contract proposal

Main ↔ renderer surface. Smaller and more explicit than today.

// ─── main → renderer broadcasts ─────────────────────────────────────────

// One channel, per-sid stream. Emits from spawn to dispose. Renderer
// attaches a Terminal as the sole consumer per sid; main does NOT gate
// emission on any "viewer count". This is what kills the buffering-mode
// dance: the consumer is always present.
channel 'pty:chunk': { sid: string; seq: number; data: Uint8Array }

// Lifecycle. Separate from chunks so consumers don't have to parse them.
channel 'pty:exit': { sid: string; code: number | null; signal: string | null }

// Sidebar's data plane. Unchanged from today, listed for completeness.
channel 'session:state': { sid: string; state: SessionState }
channel 'session:title': { sid: string; title: string }

// ─── renderer → main RPC ────────────────────────────────────────────────

// Open a SessionShell. Idempotent. Called once per sid that enters the
// sidebar; the renderer treats this as "start streaming to me". Returns
// the seq cursor the renderer should treat as the live floor.
rpc pty.open(sid): Promise<{ pid: number; cols: number; rows: number; seq: number }>

// One-shot historical snapshot. Returned snapshot is everything up to
// `seq` (exclusive). Renderer writes snapshot, then plays chunks with
// seq >= returned.seq. Replaces the 'buffering listener' indirection.
rpc pty.snapshotAt(sid, opts?: { sinceSeq?: number }): Promise<{
  snapshot: Uint8Array;
  seq: number;     // exclusive upper bound covered by `snapshot`
  cols: number;
  rows: number;
}>

// Resize is a hint; if cols/rows match current, main returns without
// emitting SIGWINCH. No-op-safe to call after every fit().
rpc pty.resize(sid, cols, rows): Promise<{ applied: boolean }>

// Input is fire-and-forget, ordered per-sid.
channel 'pty:input': { sid: string; data: string }   // renderer → main, one-way

// Drop the SessionShell. Main may keep the PTY alive (other windows /
// background) but stops emitting chunks for this sid to this window.
rpc pty.close(sid): Promise<void>

// ─── what we deliberately do NOT have ───────────────────────────────────
// • No pty.attach / pty.detach. Subscription is implicit in pty.open.
// • No getBufferSnapshot/attach split. Replaced by pty.snapshotAt, which
//   is callable at any time and returns a self-consistent {snapshot, seq}.
// • No sidebar preview channel. Sidebar reads agent state only.

Why these shapes

  • pty.open instead of pty.attach — “open” describes truth: the renderer is committing to consume this sid’s stream until pty.close. There is no transient attach/detach inside a window’s lifetime; switching sids doesn’t touch main.
  • Chunks carry seq — makes snapshot/live join trivially correct (drop chunks with seq < snapshotSeq, write the rest). No timestamp races, no “did I miss any.”
  • pty.snapshotAt is decoupled from open — the renderer can call it lazily (first time it actually needs to render history) or eagerly. The IPC is the same. Avoids the “attach implies snapshot” bundling that locks ordering.
  • resize.applied return — lets the renderer skip the “did SIGWINCH fire” heuristic; main is the source of truth.

6 · Renderer state per SessionShell

Discriminated union; transitions are the only places anything can go wrong, so they’re explicit.

type SessionShell = {
  sid: string;
  term: Terminal;            // xterm instance, created at provisioning
  container: HTMLElement;   // offscreen by default; reparented to slot on activate
  lastViewportY: number | null; // captured on each deactivate; null until first
  status: ShellStatus;
};

type ShellStatus =
  | { kind: 'provisioning' }                                   // term created, awaiting pty.open
  | { kind: 'streaming'; seq: number }                         // pty.open done, chunks flowing, no snapshot yet written
  | { kind: 'hydrating'; snapshotSeq: number }                 // snapshot fetched, currently writing
  | { kind: 'live'; baseSeq: number }                          // snapshot drained, live chunks applied directly
  | { kind: 'exited'; code: number | null }                    // pty.onExit received; terminal becomes read-only
  | { kind: 'error'; reason: ShellError; recoverable: boolean };

type ShellError =
  | 'open-failed'             // pty.open rejected (claude binary missing, spawn EPERM, etc.)
  | 'snapshot-failed'         // pty.snapshotAt rejected after retry
  | 'host-zero-size'           // rAF settled and slot is still 0×0 — layout never gave us space
  | 'stream-disconnected';    // chunk channel went silent without an exit event

// Visibility is OUTSIDE the shell — orthogonal axis:
type SlotState =
  | { kind: 'empty' }
  | { kind: 'mounting'; sid: string }       // reparented, display:'none', running Phase 2 serial tail
  | { kind: 'visible'; sid: string };       // post-reveal

The product of ShellStatus × SlotState spells out every situation. Most attach-path bugs today are illegal states in this model (e.g. visible but not yet live), which the redesign makes unrepresentable: the slot transition from mounting → visible is the same call site as display:'', and that call site requires shell.status.kind === 'live'.

7 · Failure paths

Every error has a name, a user-visible artifact, and a recovery.

FailureWhat the user seesRecovery
F1 · open-failed Slot stays empty; an inline error card replaces the xterm canvas: “Couldn’t start Claude for this session. [Retry] [View logs]”. Sidebar row gets a small error pip. Retry button calls pty.open again; on third failure we surface the claude-missing guide (existing UI).
F2 · snapshot-failed If shell never wrote a snapshot: same inline error card as F1. If shell was already live and this was a re-snapshot (rare, e.g. after dropped chunks): no UI change — we continue with current buffer and toast a non-blocking warning. Auto-retry once after 250ms; on second failure escalate to the error card.
F3 · host-zero-size Slot stays in mounting with a subtle spinner (typically <200ms). If it persists >2s: error card “Terminal area has no space — try resizing the window.” Keep re-rAFing on every ResizeObserver tick of the slot. The moment we get nonzero dims, resume Phase 2 at B3.
F4 · stream-disconnected Banner across the top of the terminal: “Live updates paused — reconnecting…”. Existing scrollback stays readable; input is disabled. Re-subscribe via pty.open; if seq jumped, run a delta snapshot via pty.snapshotAt(sid, {sinceSeq}) and apply.
F5 · exited Terminal goes read-only; ghost cursor; sidebar agent state moves to exited. A footer chip shows the exit code, with a Restart button. Restart spawns a fresh sid (re-uses the SessionShell container with a new Terminal); existing sessions never silently resurrect.

8 · Contrast with the current 19-step path

Not a diff — a structural comparison.

Current

  • 19 sequential steps in usePtyAttach.warm.ts + xtermWarmRegistry.ts, mixing data, DOM, layout, and ordering decisions.
  • Terminal lifecycle tied to attach: allocEntry on cold, display toggle on warm — different code paths.
  • Buffering-listener mode exists to bridge the gap between subscribing and being ready to render. It is correct but is a workaround for the deeper coupling.
  • Two scrollToBottom calls (L479 + L551) because the first one is structurally too early.
  • Fit happens after attach, forcing a SIGWINCH echo that arrives after the first pin (QP-3).
  • Live-tail writes are fire-and-forget (QP-1) — the highest-priority bug.
  • term.open runs in stale-or-zero layout (QP-2), CanvasAddon atlas warms up while visible (QP-4).

Redesigned

  • 5 phases, of which only Phase 2 has internal ordering — and that ordering is three independent lanes joined by one barrier.
  • One Terminal lifecycle: created at sidebar-provision, disposed at pty.close. Switching sids never touches xterm internals.
  • No buffering mode: chunks always flow to a Terminal that exists. The data plane has no “waiting room.”
  • One pin: S3. Computed in lane C, applied once, after drain, before reveal.
  • Fit runs before any resize negotiation (B3 → S1). SIGWINCH echo, if any, is absorbed in S2’s drain — never a separate event the timeline needs to worry about.
  • All writes are awaited. The drain barrier in S2 is structural, not a patched-in writeAsync('').
  • term.open runs after rAF + nonzero assert. Reveal is the last instruction, so atlas warm-up is invisible by construction.

Races that cannot occur in the new architecture

  • QP-1 (live-tail vs pin): there is no fire-and-forget tail. S2 is a single explicit drain; nothing writes between S2 and S3.
  • QP-2 (stale-dim open): term.open only runs when the container has live dims (P0 offscreen has fixed dims; B2 rAF guards the visible path).
  • QP-3 (fit-after-attach SIGWINCH echo): fit is computed before pty.resize is called, and the resize response is drained by S2.
  • QP-4 (atlas-warm in user-visible window): atlas warms up at provisioning (Phase 0) when the Terminal first opens into an offscreen container with real dims. By the time anything reveals, atlas is hot.
  • StrictMode double-mount (QP-5): shell provisioning is idempotent (keyed by sid in a renderer registry), so re-running effects is a no-op rather than a re-allocation.

Code that can be deleted

  • src/terminal/usePtyAttach.warm.ts — the 19-step procedure. Replaced by a small useVisibleShell(sid) hook that toggles SlotState.
  • xtermWarmRegistry.ts::ensureAndShowEntry / applySnapshot / pinViewportToBottom — collapse into a shellRegistry with provision/close/getSnapshot.
  • "buffering" listener mode — gone with the data-plane separation.
  • pty:attach and pty:detach IPC channels — replaced by pty.open / pty.close; the per-window stream is implicit.
  • Defensive term.reset() before snapshot — the shell registry guarantees terminals are either fresh or already live; reset has no caller.

Attach 重新设计 —— 从零开始

如果今天从头写 ccsm 的 attach 路径,它会是什么样?抛掉当前 19 步流程,以及打在上面的补丁。只谈设计。
范围:pty attach + xterm mount + 侧边栏耦合 · 配套文档 docs/attach-timeline-review.html(历史)

1 · 设计原则

后面每个决策都要回答的五条原则。

P1
第一帧 = 最终帧。
display:'' 之前,用户什么都感知不到。之后,用户必须看到一张完成的画面:滚到正确的行、atlas 已预热、尺寸为最终值。mount 到 reveal 之间步骤数无所谓;切线在哪才是关键。
P2
Terminal 是 per-sid 的长寿对象,不是 per-attach 的临时产物。
分配、term.open、addon 加载、atlas 预热、live 数据订阅,都在我们第一次有理由关心某个 sid 时发生(例如它进了侧边栏)。"Attach" 退化为让其中之一可见。切换会话从不 dispose 或重新 open xterm。
P3
数据订阅独立于可见性。
per-sid chunk 从 spawn/restore 到 dispose 持续流入 per-sid Terminal。不存在围绕 attach 切换的 "buffering listener" 模式 —— listener 永远是 Terminal 自己。可见性纯粹是 DOM 层的事。
P4
侧边栏跑在自己的数据面上。
侧边栏状态(agent 状态、标题、徽标、归档标志)来自 session:state / session:title —— 本就独立于 PTY chunk。重设计延续这一点;任何地方都不再解析 PTY 输出做最后一行预览。
P5
能并行就并行。一次 barrier。然后串行。然后 reveal。
互相独立的工作(snapshot 拉取、布局 commit、addon 预热)在不可见空间内并发跑。一次 await Promise.all 汇合。串行尾部只做真正有顺序约束的事(写 snapshot → drain live tail → 钉住 viewport)。Reveal 是最后一条指令。

2 · 总体架构

五个阶段。从左往右读。

Phase 0
Provision
sid 进入侧边栏时,在 renderer 侧分配一个 SessionShell:offscreen 容器里的 xterm Terminal、addon 已加载、live 流订阅已开启。无用户可见副作用。
Phase 1
持续 stream
PTY chunk 持续落入 per-sid Terminal。侧边栏独立接收 session:state。两条线永远在跑,与当前激活的是哪个 sid 无关。
Phase 2
准备可见
用户选中某个 sid。并行:把 SessionShell reparent 到可见宿主(仍 display:'none')、commit 布局、按真实尺寸 fit、恢复上次 viewport(warm)或贴底(cold)。
Phase 3
Reveal
一条赋值:display:''。viewport、atlas、尺寸、内容 —— 都已为最终态。这是用户看到的唯一一帧。
Phase 4
Reveal 之后
接通输入路由(在 Terminal 层早已接好,只翻一个路由标志)、focus、在 store 标记 active、安装/刷新 ResizeObserver 基线。

为什么是这个形状

当前路径把四件正交的事 —— Terminal 生命周期、数据订阅、布局、可见性 —— 揉成一个叫 "attach" 的大过程。把它们解耦,所有重活都从点击路径中移走。等用户点会话时,字节已经流了几秒;"attach" 退化为 "show"。

3 · Phase 2 内部的并行流

仍在 display:'none' 时可以并发跑什么、什么必须串行、reveal 落在何处。

Lane A · 数据
A1. 若 shell 尚未写过 snapshot,调 pty.snapshotAt(sid, sinceSeq?);否则跳过(已 live)。
A2. writeAsync(snapshot) 写入 Terminal scrollback。Drain。
A3. Drain 截至当前的 in-flight live chunk(一次显式 await term.write('', cb) barrier —— 全流程唯一使用处)。
Lane B · DOM
B1. 把 SessionShell 从 offscreen pool reparent 到可见 <TerminalSlot/>,仍 display:'none'。
B2. await rAF —— 让布局 commit。断言 clientWidth/Height > 0(否则进入错误路径 F3)。
B3. fit.fit() → 按真实尺寸算出最终 cols/rows。
Lane C · Viewport 计划
C1. 读 shell 的 lastViewportY(warm)或 null(cold)。
C2. 决定目标:warm → 恢复 Y;cold+resume → 贴底;cold+空 → 贴底(no-op)。
C3. 借 shell 的 scrollback 行数预算;此时不修改 terminal。
▼
await Promise.all([ A, B, C ]) 单点 barrier —— 三条 lane 全部就绪
▼
串行尾部 · 仍 display:'none'
S1. 若 fit.cols ≠ pty.cols:pty.resize(sid, cols, rows)。引发的 SIGWINCH 回显落在 live 流(已指向该 Terminal);不重新拉 snapshot —— parser 会追上。
S2. 一次 await term.write('', cb) drain —— 同时吸收 A3 的 live tail和 S1 的 SIGWINCH 回显。
S3. 应用 lane C 的 viewport 计划:term.scrollToLine(targetY) 或 term.scrollToBottom()。一次。精确。
▼
slot.style.display = '' 用户看到的第一帧 —— 已是最终态
▼
Phase 4 · reveal 之后 · 无序
P1. 在 store 翻转 activeSid → 输入路由开始把击键送到该 sid 的 pty.input。
P2. term.focus()。
P3. 安装 ResizeObserver 基线(吃掉 display 翻转引起的第一个伪 tick,与当前实现同理)。

依赖说明

  • A 与 B/C 独立。A 可以在 B reparent 之前完成 —— 写入会进入一个尚未上屏的 Terminal,没问题(Phase 1 / P3)。
  • B2(rAF)是唯一 DOM 时序 barrier。便宜、可预测:至多一帧,通常 <16ms。
  • C 是纯计算。独立成一条 lane,是为了把 warm/cold 决策作为 S3 的显式数据输入,而非藏在 S3 里的分支泥潭。

4 · 各场景下的第一帧保证

从用户视角看只存在两种场景。各自看到什么,以及保证它成立的不变量。

场景 A —— cold + resume有历史的会话首次 attach

第一帧:最终尺寸的完整 xterm 画布,scrollback 渲染到 resume snapshot 的最后一行,滚动条贴底,光标停在 CLI 离开的位置。无空帧、无向上漂移、无延迟 snap。
  • 怎么做到:snapshot 写入(A1–A2)和 live-tail drain(A3、S2)都在 reveal 之前完成。Viewport 目标在 C 计算(贴底),在 S3 一次性应用。Reveal 在 S3 之后。
  • 为何无 race:没有 fire-and-forget 的写。S2 的单点 drain barrier 是 reveal 前 parser 队列可能非空的唯一位置,我们显式 await 它。
不变量:display:'' 那一刻,term.buffer.baseY === lastWrittenBaseY 且 ydisp === baseY。两者由 S2(无 pending 写入)和 S3(恰好一次钉位)共同保证。

场景 B —— warm 切换之前 attach 过,切回

第一帧:用户离开时的精确 viewport 位置,包括 scrollback 中间位置。滚动条对齐。脱离期间到达的新 chunk 在 viewport 下方的 scrollback 中(不会被自动滚到)。
  • 怎么做到:SessionShell 从未被 dispose(P2)。Lane A 是 no-op —— chunk 一直流入这个 Terminal。Lane C 读取已持久化的 lastViewportY(每次失去可见性时记下)。S3 调 scrollToLine(targetY)。
  • Reparent 考虑:reparent(B1)不会重置 xterm 内部状态。仍走 rAF(B2)gate,以防宿主之间宽度变化。
不变量:sid X 的 renderer 状态在 deactivate 之前与 reactivate 之后相同,除了 baseY 可能因 live chunk 前进过(这是正确的 —— 它们应在 buffer 里,只是不拉动 viewport)。

5 · IPC 契约提案

主进程 ↔ renderer 接口面。比现在更小、更显式。

// ─── main → renderer 广播 ───────────────────────────────────────────────

// 单一通道,per-sid 流。从 spawn 到 dispose 都在 emit。renderer 以
// Terminal 作为该 sid 的唯一消费者;主进程不按 "viewer count" gate
// emit。这正是杀掉 buffering-mode 把戏的关键:消费者永远在线。
channel 'pty:chunk': { sid: string; seq: number; data: Uint8Array }

// 生命周期。与 chunk 分开,消费者无需解析 chunk。
channel 'pty:exit': { sid: string; code: number | null; signal: string | null }

// 侧边栏数据面。与现状一致,此处列出以求完整。
channel 'session:state': { sid: string; state: SessionState }
channel 'session:title': { sid: string; title: string }

// ─── renderer → main RPC ────────────────────────────────────────────────

// 打开一个 SessionShell。幂等。每个进入侧边栏的 sid 调用一次;
// renderer 视其为 "开始 stream 给我"。返回 renderer 应视为 live floor
// 的 seq 游标。
rpc pty.open(sid): Promise<{ pid: number; cols: number; rows: number; seq: number }>

// 一次性历史 snapshot。返回的 snapshot 覆盖到 `seq`(不含)之前的全部
// 内容。renderer 写入 snapshot,再播放 seq >= returned.seq 的 chunk。
// 替代 'buffering listener' 这一层间接。
rpc pty.snapshotAt(sid, opts?: { sinceSeq?: number }): Promise<{
  snapshot: Uint8Array;
  seq: number;     // `snapshot` 覆盖的上界(不含)
  cols: number;
  rows: number;
}>

// Resize 是 hint;若 cols/rows 与当前一致,主进程返回但不 emit SIGWINCH。
// 每次 fit() 后无副作用地调用都是安全的。
rpc pty.resize(sid, cols, rows): Promise<{ applied: boolean }>

// 输入是 fire-and-forget,per-sid 有序。
channel 'pty:input': { sid: string; data: string }   // renderer → main,单向

// 丢弃该 SessionShell。主进程可能保留 PTY 存活(其他窗口 / 后台),
// 但停止向本窗口 emit 该 sid 的 chunk。
rpc pty.close(sid): Promise<void>

// ─── 我们刻意不提供的东西 ───────────────────────────────────────────────
// • 没有 pty.attach / pty.detach。订阅隐含在 pty.open 里。
// • 没有 getBufferSnapshot/attach 拆分。由 pty.snapshotAt 替代,
//   可在任意时刻调用,返回自洽的 {snapshot, seq}。
// • 没有侧边栏预览通道。侧边栏只读 agent 状态。

为何是这些形状

  • pty.open 而非 pty.attach —— "open" 更贴合事实:renderer 承诺消费该 sid 的流直到 pty.close。窗口生命周期内不存在瞬态 attach/detach;切换 sid 不触及主进程。
  • chunk 携带 seq —— 让 snapshot/live 拼接天然正确(丢弃 seq < snapshotSeq 的 chunk,其余写入)。无时间戳竞争,无 "是否漏了"。
  • pty.snapshotAt 与 open 解耦 —— renderer 可以懒调(首次真正需要渲染历史时)或主动调,IPC 形状一致。避免 "attach 即 snapshot" 的捆绑锁死顺序。
  • resize.applied 返回值 —— 让 renderer 跳过 "SIGWINCH 是否触发了" 的猜测;主进程是事实源。

6 · 每个 SessionShell 的 renderer 状态

Discriminated union;转移是唯一可能出错的地方,所以显式列出。

type SessionShell = {
  sid: string;
  term: Terminal;            // xterm 实例,在 provision 时创建
  container: HTMLElement;   // 默认 offscreen;activate 时 reparent 到 slot
  lastViewportY: number | null; // 每次 deactivate 时记录;首次之前为 null
  status: ShellStatus;
};

type ShellStatus =
  | { kind: 'provisioning' }                                   // term 已创建,等待 pty.open
  | { kind: 'streaming'; seq: number }                         // pty.open 完成,chunk 流动,尚未写过 snapshot
  | { kind: 'hydrating'; snapshotSeq: number }                 // snapshot 已拉,正在写入
  | { kind: 'live'; baseSeq: number }                          // snapshot 已 drain,live chunk 直接应用
  | { kind: 'exited'; code: number | null }                    // 收到 pty.onExit;terminal 转为只读
  | { kind: 'error'; reason: ShellError; recoverable: boolean };

type ShellError =
  | 'open-failed'             // pty.open 被拒(claude 二进制缺失、spawn EPERM 等)
  | 'snapshot-failed'         // pty.snapshotAt 重试后仍失败
  | 'host-zero-size'           // rAF 已 settle,slot 仍为 0×0 —— 布局未给空间
  | 'stream-disconnected';    // chunk 通道静默,但未收到 exit 事件

// 可见性在 shell 之外 —— 正交轴:
type SlotState =
  | { kind: 'empty' }
  | { kind: 'mounting'; sid: string }       // 已 reparent,display:'none',跑 Phase 2 串行尾部
  | { kind: 'visible'; sid: string };       // reveal 之后

ShellStatus × SlotState 的笛卡尔积穷尽了所有情况。当前 attach 路径的多数 bug 在这个模型里都是非法态(例如已可见但还没 live),重设计让它们无法表达:mounting → visible 的 slot 转移与 display:'' 是同一个调用点,而该调用点要求 shell.status.kind === 'live'。

7 · 失败路径

每个错误都有名字、用户可见表现、恢复方式。

失败用户看到什么恢复
F1 · open-failed Slot 保持空;内联错误卡替代 xterm 画布:"无法为该会话启动 Claude。[重试] [查看日志]"。侧边栏行带小错误标点。 重试按钮再次调 pty.open;第三次失败时弹出 claude-missing 指引(沿用现有 UI)。
F2 · snapshot-failed 若 shell 从未写过 snapshot:同 F1 的内联错误卡。若 shell 已 live、这次是重新 snapshot(罕见,如 chunk 丢失之后):无 UI 变化 —— 继续沿用当前 buffer,toast 非阻塞警告。 250ms 后自动重试一次;第二次失败升级为错误卡。
F3 · host-zero-size Slot 停留在 mounting 并显示淡 spinner(通常 <200ms)。若超过 2s:错误卡 "终端区域无空间 —— 试试调整窗口大小。" 每次 slot 的 ResizeObserver tick 都重排 rAF。一旦拿到非零尺寸,从 B3 恢复 Phase 2。
F4 · stream-disconnected 终端顶部横幅:"实时更新已暂停 —— 重连中…"。已有 scrollback 仍可读;输入禁用。 通过 pty.open 重新订阅;若 seq 跳跃,调 pty.snapshotAt(sid, {sinceSeq}) 做增量 snapshot 并应用。
F5 · exited 终端转只读;光标变灰;侧边栏 agent 状态变为 exited。底部 chip 显示退出码,带重启按钮。 重启 spawn 一个新 sid(复用 SessionShell 容器,搭配新的 Terminal);已有会话不会静默复活。

8 · 与当前 19 步路径的对比

不是 diff,是结构性对比。

当前

  • 19 个串行步骤分布在 usePtyAttach.warm.ts + xtermWarmRegistry.ts,数据、DOM、布局、顺序决策混在一起。
  • Terminal 生命周期绑死 attach:cold 走 allocEntry,warm 走 display toggle —— 两条不同代码路径。
  • buffering-listener 模式用来弥合订阅与可渲染之间的空档。它是正确的,但本质是更深层耦合的 workaround。
  • 两次 scrollToBottom(L479 + L551),因为第一次结构上太早。
  • fit 在 attach 之后发生,导致 SIGWINCH 回显出现在第一次钉位之后(QP-3)。
  • live-tail 写入是 fire-and-forget(QP-1)—— 最高优先级的 bug。
  • term.open 在过期或零尺寸布局中执行(QP-2),CanvasAddon atlas 在可见状态预热(QP-4)。

重设计

  • 5 个阶段,只有 Phase 2 有内部顺序 —— 且这个顺序是三条独立 lane 加一次 barrier。
  • 单一 Terminal 生命周期:侧边栏 provision 时创建,pty.close 时 dispose。切换 sid 永远不触碰 xterm 内部。
  • 无 buffering 模式:chunk 永远流向一个已存在的 Terminal。数据面没有 "等候室"。
  • 一次钉位:S3。在 lane C 算好,drain 后、reveal 前一次应用。
  • Fit 在任何 resize 协商之前发生(B3 → S1)。SIGWINCH 回显(若有)被 S2 的 drain 吸收 —— 时间线不需要单独操心。
  • 所有写入都被 await。S2 的 drain barrier 是结构性的,不是补丁式的 writeAsync('')。
  • term.open 在 rAF + 非零尺寸断言之后运行。Reveal 是最后一条指令,所以 atlas 预热在结构上不可见。

在新架构下不可能发生的 race

  • QP-1(live-tail vs 钉位):不存在 fire-and-forget tail。S2 是单点显式 drain;S2 与 S3 之间无任何写入。
  • QP-2(stale-dim open):term.open 只在容器有 live 尺寸时运行(P0 offscreen 有固定尺寸;B2 rAF 保护可见路径)。
  • QP-3(fit-after-attach SIGWINCH 回显):fit 在 pty.resize 之前算好,resize 的响应被 S2 drain。
  • QP-4(用户可见窗口内 atlas 预热):atlas 在 provision(Phase 0)即预热 —— 此时 Terminal 首次 open 到带真实尺寸的 offscreen 容器。等任何东西 reveal 时,atlas 已热。
  • StrictMode 双 mount(QP-5):shell provisioning 是幂等的(在 renderer registry 中以 sid 为键),所以 effect 重跑就是 no-op,不会重新分配。

可以删掉的代码

  • src/terminal/usePtyAttach.warm.ts —— 19 步过程。由一个小 useVisibleShell(sid) hook 取代,只翻 SlotState。
  • xtermWarmRegistry.ts::ensureAndShowEntry / applySnapshot / pinViewportToBottom —— 折叠成 shellRegistry,只暴露 provision/close/getSnapshot。
  • "buffering" listener 模式 —— 随数据面解耦一并消失。
  • pty:attach 与 pty:detach IPC 通道 —— 由 pty.open / pty.close 取代;per-window 流是隐式的。
  • snapshot 前的防御性 term.reset() —— shell registry 保证 terminal 要么全新,要么已 live;reset 无人调用。

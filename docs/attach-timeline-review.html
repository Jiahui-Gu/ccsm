Why so many steps? — 历史包袱 + 真正可砍 + 需要更深改造

逐步审视当前 19 步,分三类。

看起来多余 · 但有真实原因(保留语义,可简化实现)

buffering listener mode (原 3)
不是绕过老 bug——是 Major 1 fix from cold review(commit fbec902c, PR #1355)的核心架构。listener 在 alloc 时立即装,这样 alloc → attach IPC resolve → snapshot 写入这段窗口里到达的 chunks 不会丢、不会被 reset() 抹掉、也不会和 snapshot 重复。并行设计下尤其必须:Block C 比 Block A 先 ready 是常态。
term.reset() before snapshot write (原 10)
代码注释自己说"purely defensive"——entry 来自 allocEntry,Terminal 是新的,buffer 应该已经是空的。逻辑上可删,但风险点:retry 路径 / 重入 / cancel-then-resume 时如果 entry 复用,term 可能已被写过。代价是一次 O(rows) 的 buffer 清空,比维护"reset 真的不必要"的不变量便宜。保留,但归在 S3 内合并。
两次 scrollToBottom(L479 + L551) (原 13 + 17)
来自 PR #1352(commit 4b5c6d9e,"viewport-pinning invariant — eliminate scroll-to-top race on idle re-attach")。L479 是 snapshot 写完立刻 pin,L551 是 fit + 可能的 SIGWINCH 重发之后再 pin 一次。本质是补偿前一次跑得太早。并行设计把 fit 提前 + drain 改 await 之后,只剩一次就够。原因合理,但可以合并。
ResizeObserver baseline first-tick skip (原 19)
因为 display:'none'→'' 自带一次 spurious RO callback(参考 PR #1367 dogfood frame log:viewportY 294→0,baseY 344→0)。如果不吞掉这一次会触发不必要的 snapshot replay。必须保留,但和 attach 时序解耦,可以挪去 P4。

真正可砍 / 可合并

writeAsync('') 占位 drain in pinViewportToBottom (helper L86-97)
只是给 fire-and-forget 的 live-tail write 当 drain barrier 用——是 fix QP-1 的临时拐杖。S3 把 live-tail 改成显式 await drain 之后,这个空 write hack 整体可删。整个 pinViewportToBottom 可删除。
L514 的 fit + 条件 pty.resize (原 15)
当前在 snapshot 写完之后才跑,顺序错。提前到 S1(term.open 之后立刻),并且 pty.resize 改 fire-and-forget(原 8 的 term.resize cached-estimate 整步可砍)。原 8 整步删除,原 15 合并进 S1+S2。
两次 IPC(pty.attach + getBufferSnapshot)合并 (原 7 + 9)
两次 IPC 之间 main 已在广播 chunks。合并成一个 IPC 返回 {cols, rows, snapshot, seq} 后,buffered 窗口最小化、renderer await 数 -1。需要 main process 改造,但语义干净。
L479 scrollToBottom (原 13)
和 L551 重复。drain 全 await 之后只需要 reveal 前那一次。

需要更深改造(超出本 PR 范围)

main process IPC: pty.attachAndSnapshot — A1 合并需要改 main side。短期可在 renderer 侧用 Promise.all([pty.attach(sid), Promise.resolve()]) 然后 await snapshot 模拟并行,但 main process 仍是两次 round-trip,buffered 窗口没缩短。建议:先 renderer-only 重构,验证视觉效果;main 侧 IPC 合并做后续 task。
P4 ResizeObserver 装载时机 — 理论上可以挪到 hook 初次 mount 时就装好,而不是 attach 完成之后。但 RO 的 baseline 必须发生在 wrapper 第一次 reveal 之后(才能拿到真实 contentRect),所以这条路要重设计 baseline 协议。低优。

真正可疑 · 推荐深入调查

原步 10 term.reset() 的必要性 — 代码注释自己写"purely defensive (term is fresh from allocEntry)"。如果不变量真成立,这步根本不该存在;如果不变量不成立,那是 allocEntry / retry / cancel 路径的 bug,应该修不变量而不是每次 attach 都 reset。necessity unclear, recommend deeper investigation——值得一个独立 task 验证 entry 在所有进入 S3 的路径下是否保证 buffer 为空,以及"复用半完成 entry"是否真的可能(代码看起来 cancel 路径会 disposeEntry,但 retry 路径不会 dispose)。
设计哲学 — 用户看到的第一帧 = 定稿。
reveal (display:'') 之前的所有事用户感知不到,所以"步数"不是优化目标。 原则:① 没依赖关系的工作并行做(PTY IPC / layout / entry alloc 三条线同时跑); ② 一个总 await Promise.all 兜底,所有准备 settled 后才进入串行收尾; ③ 串行收尾只做有依赖关系的最少动作(open+fit → reset+write+drain → scrollToBottom); ④ 最后一刻 reveal,用户首帧就是定稿; ⑤ wireup / focus / setState / ResizeObserver baseline 这类用户感知不到的事放 reveal 之后即可,顺序无关紧要。
核心改动一句话:把 fit + drain 全部提前到 reveal 之前,只在底部对齐一次后再翻 display:'',让用户看到的"第一帧"就是定稿。
  • 顺序变更:fit.fit 移到 pty.attach 之前(原 L514 → 提前到 ~L410 之前)
  • 新增 await:applySnapshot 的 live-tail term.write 改为带 cb 的 Promise drain(原 L990 fire-and-forget → await)
  • 新增 await:term.open 之前 rAF + 0×0 guard,等 layout commit
  • 合并:L479 + L551 两次 scrollToBottom → 一次(放在 drain 完之后)
  • 移动:wrapper display:'' 由 alloc 后(L844)推到 bottom-pin 之后
  • 删除:pinViewportToBottom 整体(及 writeAsync('') 占位 drain)

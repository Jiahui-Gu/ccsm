## Per-item verdicts

### 1. Descriptor `boot_id` + atomic write (F2 / chapter 03 §3) — **PASS**

Evidence:

- Atomic write discipline pinned in `03-listeners-and-transport.md` §3.1: write `listener-a.json.tmp` → `fsync(2)` → `rename(2)` (or `MoveFileExW(MOVEFILE_REPLACE_EXISTING)` on Windows). All three steps named explicitly; rename atomicity invariant called out.
- `boot_id` field present in the descriptor JSON example (line 132) and field reference (line 146): random UUIDv4 per daemon boot, regenerated on every daemon process start, held in memory for the daemon's lifetime.
- Electron-side staleness rejection pinned in `03-listeners-and-transport.md` §3.3: Electron calls `Hello` immediately on connect; if descriptor `boot_id != Hello.boot_id`, Electron MUST close the connection, discard the in-memory descriptor, re-read from disk, and retry. Reconnect path also re-reads the file.
- Supervisor `/healthz` echoes the same `boot_id` (line 228) so operators / Electron main can correlate.
- Cross-chapter restate in `07-data-and-state.md` §2 (lines 44–50) confirms the atomic-write + per-boot-rewrite + orphan-files-OK invariants.

The original R2 P0-02-3 / P0-03-4 / P0-07-1 cluster is mechanically closed. Wording is unambiguous.

### 6. Update flow specified (F9 / chapter 10 §8) — **PASS**

Evidence:

- `10-build-package-installer.md` §8 (lines 174–183) pins the four-step stop-replace-restart-rollback flow:
  1. Stop service (per-OS commands) with 10 s timeout, then `Stop-Service -Force` / SIGKILL escalation, verified by `Get-Process` / `pgrep`.
  2. Replace binary via atomic rename (existing → `.prev`; staging → in-place). Same-volume invariant called out so rename is atomic on every OS. `native/` directory ACLs preserved.
  3. Restart service (per-OS commands).
  4. Health check via Supervisor `/healthz` for HTTP 200 within 10 s; on timeout → rollback (atomic rename of `.prev` back, restart, re-poll, log `crash_log` source `update_rollback`).
- Operator visibility: rollback always logs to `crash_log` so the user sees update failure surfaced via chapter 09. Updater UX surfaces both update-success and update-rollback states.
- Smoke test: `tools/update-flow.spec.{ps1,sh}` exists as a manual pre-release smoke (CI promotion deferred to v0.4 — acceptable for v0.3 dogfood).
- §8 explicitly notes the flow trusts upstream `verify-signing.*` (no double-verification). Boundary clean.

R2 P0-10-1 closed.

### 7. DNS-rebinding (bridge UDS / per-OS socket on bridge↔daemon hop) (F2 / chapter 08 §4.2) — **PASS** (with the dispatch-plan §0 caveat)

Per dispatch-plan §0: only the **structural** DNS-rebinding fix is in scope; per-request `Host:` allowlist + bearer token are demoted to v0.4. Two structural defences shipped:

Evidence:

- Bridge↔daemon hop: `08-electron-client-migration.md` line 205 — bridge speaks the daemon's chosen Listener A transport (UDS / named pipe / loopback TCP / loopback TLS). For UDS / named pipe the bridge is the ONLY caller; the renderer never touches the OS-level socket. F2 close-out comment (line 152) restates: "DNS rebinding mitigated by bridge bound to UDS / named pipe (no loopback TCP for bridge↔daemon)". This is the structural ask: there is no TCP socket on the bridge↔daemon hop for a browser tab to rebind to.
- Renderer↔bridge hop: line 204 — `http2` server bound on `127.0.0.1` only (no `0.0.0.0`); `Host:` header MUST equal `127.0.0.1:<our-port>` else 421 Misdirected Request. This is the "structural part" the dispatch-plan §0 explicitly kept in scope; bearer-token belt-and-suspenders is the explicitly-deferred v0.4 work.
- Descriptor authenticity layer (line 152, 190): renderer constructs Connect transport from descriptor and runs `Hello`-echo `boot_id` verification before any other RPC.

Caveat for the audit trail: the renderer↔bridge hop is loopback TCP (Chromium fetch cannot speak UDS). The chapter is honest about this and the `Host:` allowlist + 421 response is the documented structural mitigation. This matches the dispatch-plan §0 scope statement word-for-word; not a gap.

R2 P0-08-1 (structural) closed.

### 8. PID-recycling (covered by descriptor + UDS choices) — **PASS**

Evidence:

- `03-listeners-and-transport.md` line 147 explicitly documents the design intent: `daemon_pid` in the descriptor is "observability only (Electron does NOT use this for auth — pids recycle)". The `boot_id` mismatch path (§3.3) is the trust witness, not the pid.
- Supervisor is UDS-only on every OS (item 2 above) → peer-cred derives uid via `SO_PEERCRED` / `LOCAL_PEERCRED` / named-pipe SID (kernel-supplied, no PID-table lookup). The TOCTOU/PID-recycle window that round-1 P0-03-2 worried about does not exist on the supervisor surface.
- Listener A: §5 lines 200, 204 — UDS gives free peer-cred via `getsockopt(SO_PEERCRED)` / `LOCAL_PEERCRED`. The loopback-TCP variant (A2) does perform PID lookup, but the daemon trust hinge has been moved off pid (auth) onto kernel-supplied uid (peer-cred) + boot_id (transport identity). PID lookup A2 is now an OS-fallback only and the residual TOCTOU window is acknowledged in the descriptor schema (`daemon_pid` not used for auth) and in the spike registry (`[security-pid-recycle-tcp-peercred]` in chapter 14, demoted-but-tracked).
- No place in the reviewed chapters uses PID as a trust token without the descriptor handshake. Crash-log `daemon-self` sentinel (chapter 09) and per-session attribution (chapter 05) both go through `principalKey` / peer-cred uid, never raw pid.

R2 P0-03-2 (structural) closed.

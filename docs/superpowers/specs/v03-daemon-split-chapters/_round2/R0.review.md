### P1-NEW-1 — `OwnerFilter`/`SettingsScope`/`WatchScope` v0.3 acceptance semantics not consistent across chapters 04 and 05 (soft contradiction, not zero-rework-blocking)

**Location**: `04-proto-and-rpc-surface.md` §5 final paragraph vs `05-session-and-principal.md` §5 row "GetCrashLog / WatchCrashLog".

- Chapter 04 §5 closing paragraph: "v0.3 daemon enforces only that `OWNER_FILTER_ALL` is rejected for non-`local-user` principals; v0.4 adds full per-principal scoping additively".
- Chapter 05 §5 enforcement-matrix row: "`OWNER_FILTER_ALL` is permitted in v0.3 only because there is exactly one local-user principal whose effective view is identical to `ALL`".
- Chapter 04 §6 (Settings): "v0.3 daemon rejects `SETTINGS_SCOPE_PRINCIPAL` with `InvalidArgument` (the enum value exists for v0.4 only)" — this one is unambiguous.
- Chapter 05 §5 (`WatchSessions` row): "`WATCH_SCOPE_ALL` is rejected with `PermissionDenied` in v0.3" — also unambiguous.

The asymmetric "ALL is permitted in v0.3 for local-user only" rule for `OwnerFilter` is consistent across both files but is a third behavior pattern alongside `SettingsScope` (always rejected) and `WatchScope` (always rejected). A v0.4 contributor reading chapter 15 §3 forbidden-pattern 14 ("flipping defaults, adding behavior to existing enum values, or reading scope from any source other than the request enum is a hard block") could plausibly re-read this as "v0.3 daemon already does branch on `ctx.principal.kind` for `OWNER_FILTER_ALL` — therefore `WATCH_SCOPE_ALL` should too". That is the soft additivity hazard.

**Why P1, not P0**: the wire shape and the column/table shape are correct and locked; v0.4 adding admin gating for all three enums is purely additive (daemon-side branch on principal kind). The risk is a contributor *reshaping* the v0.3 acceptance rule, not the wire schema.

**Suggested fix (v0.3, before ship)**: pick one rule for all three enums and document in chapter 15 §3 forbidden-pattern 14. Recommendation: **all three enums reject the `ALL` / `PRINCIPAL` value with `PermissionDenied` (or `InvalidArgument`) in v0.3; v0.4 admin principal kind unlocks them**. Update chapter 04 §5 closing paragraph + chapter 05 §5 `GetCrashLog` row to match.

### P1-NEW-2 — Ship-gate (b) test step 6 still embeds the "no snapshot frame" retention coupling (R1 P1.5 only partially closed)

**Location**: `12-testing-strategy.md` §4.2 step 6.

R1 P1.5 said: rephrase the assertion as "If 100 deltas occurred during the gap (which is < 4096 retention), reattach receives those 100 deltas without a snapshot". F12 added the load-bearing byte-equality assertion at step 7 — that is the gate that actually catches data loss — but step 6's text still asserts "no `snapshot` frame (still within retention window)" verbatim from the R1-flagged shape. If v0.4 tunes `DELTA_RETENTION_SEQS` from 4096 down for memory reasons, step 6 fails for non-architectural reasons.

**Why P1, not P0**: step 7 (byte-equality) is the load-bearing assertion; step 6 failing for a retention-window reason is a CI flake, not a v0.4 design rework. v0.4 could address by updating the test alongside the retention tune (additive PR; not a zero-rework violation).

**Suggested fix**: rephrase step 6 to derive from the configured retention rather than embedding "no snapshot frame ever".

### P1-NEW-3 — `assertOwnership` signature retained (R1 P1.1 chapter 05 not closed) — soft additivity hazard for v0.4 capability-based access

**Location**: `05-session-and-principal.md` §4 unchanged.

R1 P1.1 suggested reframing `assertOwnership(p, s)` as `assertAccess(p, s, capability)` from v0.3 onward so v0.4 access policies (delegation, shared-team sessions) plug in without an `if` chain. The current spec keeps `assertOwnership` and accepts the early-return chain. Per chapter 15 §2 row [05 §4], v0.4 admin clause is documented as additive ("gains optional admin clause; existing logic unchanged").

**Why P1, not P0**: the function lives in `packages/daemon/src/auth.ts` and is daemon-internal — its signature change in v0.4 would touch only daemon source files, NOT proto / schema / installer. The forbidden-pattern list does not protect it. So strictly per the rubric this is acceptable; flagged here as a soft refactor risk.

**Suggested fix**: optional. Either accept (current state, acceptable per strictness gradient) OR add a chapter 15 §3 forbidden-pattern "v0.4 MUST NOT change the `assertOwnership` signature; new access-policy variants ship as a sibling `assertAccess(p, s, capability)` function that calls `assertOwnership` for the legacy capability". Manager can decide.

### P1-NEW-4 — Chapter 09 §6 macOS hang detection is now explicitly v0.4 hardening, but the brief §11(c) ship-gate runs on all 3 OSes and could mask a macOS daemon hang

**Location**: `09-crash-collector.md` §6 + `14-risks-and-spikes.md` §1.15 (REMOVED from spike register).

F11 explicitly removed `[watchdog-darwin-approach]` from the v0.3 spike register and documented "macOS hang detection is deferred to v0.4 hardening". **Why P1, not P0**: the deferral is purely additive — v0.4 watchdog adds without touching any v0.3 code/proto/schema/installer. The risk is dogfood-quality during v0.3 soak on macOS, not zero-rework.

**Suggested fix**: optional. Add a sentence to `12-testing-strategy.md` §4.3 or §7 noting that on macOS a stalled-stream observed during the soak is interpreted as a daemon hang (no kernel watchdog will reap it).

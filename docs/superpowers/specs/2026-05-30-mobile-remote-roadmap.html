1 · 现在的架构(merged 之后的真实形态)

关键事实:mobile remote 直接和 ptyHost 对话,完全绕过桌面用的 window.ccsm bridge 层。这是理解后续所有工作的前提。

  ┌─────────────────────────────┐         ┌──────────────────────────────────────┐
  │   桌面 Renderer (src/)       │         │        手机浏览器 (mobilePage.ts)       │
  │   React + xterm.js          │         │        内联 HTML + xterm.js (CDN)       │
  └──────────────┬──────────────┘         └───────────────────┬────────────────────┘
                 │ window.ccsm                                 │ WebSocket  /ws?token=…
                 │ (6 preload bridges, IPC)                    │ (token-gated, 127.0.0.1)
                 ▼                                             ▼
  ┌─────────────────────────────┐         ┌──────────────────────────────────────┐
  │  preload/bridges/*  ──IPC──▶ │         │  electron/remote/                     │
  │  ipc/*Ipc.ts handlers        │         │  mobileRemoteServer · remoteMessages   │
  └──────────────┬──────────────┘         │  wsProtocol · remoteHttp · mobilePage  │
                 │                         └───────────────────┬────────────────────┘
                 │                                             │ 直接 import
                 ▼                                             ▼
        ┌──────────────────────────────────────────────────────────────┐
        │                    electron/ptyHost/  (node-pty)               │
        │  listPtySessions · inputPtySession · resizePtySession          │
        │  getBufferSnapshot · onPtyData(seq)  ←── 单一真相源 (seq 权威)   │
        └──────────────────────────────────────────────────────────────┘

  两条路径平行、各自直连 ptyHost。WS 这条是新加的,没有复用 IPC 的契约。
设计张力:桌面和手机各写了一份"会话协议"。桌面用 IPC channel 常量 + 类型;手机用 JSON over WS。两边其实在表达同一组操作(list / snapshot / input / resize / data 流)。这就是 LATER transport 抽象想统一的东西。

3 · 工作分解(候选,按风险/价值排)

A · NOW 真机打磨 — 风险 低 · 价值 最直接

headless 测不到的 OS 级交互,只有真手机能暴露。这是"真正手机能用"的最后一公里。

工作项说明风险
软键盘遮挡iOS/Android 弹出软键盘时,终端可视区被压缩 — 用 visualViewport API 调 #terminal 高度低
横竖屏切换旋转后 FitAddon 重算 + resize 防抖,验证不丢行低
触控手感滚动 vs 选择冲突;长按粘贴;keybar 按钮命中区 ≥44px(已 set,需真机确认)中
tailscale 实连时延真机经 tailscale 连,测输入回显延迟 + 重连行为低
PWA / 添加到主屏(可选)加 manifest,让手机像 app 一样全屏打开低
需要你配合:这一档大部分要你拿真手机连一次、点一点、录个屏 —— headless 给不了"手感"证据。我可以先把 visualViewport/旋转这些能在浏览器 devtools 模拟的做掉,留触感类给你真机验收。

B · NEXT 多客户端 / 状态健壮性 — 风险 中 · 价值 中

WS 心跳 ping/pong现在 server 不发 ping(代码注释已标),手机锁屏/切后台后死连接不会被清。加心跳 + 超时回收中
多手机同看一个 session当前架构支持,但没测过两个 client 订阅同一 sid 的 snapshot/seq 一致性中
断线缓冲手机断网期间的 pty 输出靠重连后 snapshot 补 —— 验证大输出场景不丢/不卡中

C · LATER Transport 抽象 — 风险 高 · 价值 高(长期)

原 brainstorm 的"正确"方向:让桌面和手机共用一套会话协议契约。动 6 个 preload bridge,桌面回归风险高,不该无人值守落。

  目标形态:抽出一个 "SessionTransport" 契约(list/snapshot/input/resize/data 流),
            桌面用 IPC adapter 实现,手机用 WS adapter 实现,两边共享类型 + 校验。

  2-phase 设计(原始 brainstorm):
    phase-1  协议契约重构 — 把会话操作抽成与传输无关的 interface + 共享类型/校验
    phase-2  WS adapter — mobile remote 改成实现该 interface,删掉重复的 JSON 协议手写代码

  风险点:phase-1 要碰 ccsmPty / ccsmSession 两个 bridge 和对应 IPC handler,
          桌面终端是核心路径,任何回归都是 user-visible。必须真机 + dogfood 双证据。

4 · 我的推荐顺序

建议:A(真机打磨可模拟的部分) → B(心跳/多端) → 再决定 C。
  • A 价值最直接、风险最低,且是"真正手机能用"的字面意义。我能无人值守做掉 visualViewport 软键盘、横竖屏、PWA manifest 这些 devtools 可验证的;触感类标记为"待你真机验收"。
  • B 心跳是真实健壮性缺口(死连接不回收),风险可控,值得紧接着做。
  • C transport 抽象价值高但碰桌面核心路径,等你醒来一起定方向再动 —— 符合"高风险不无人值守落"的约束。
  • D 是平行的技术债,任何时候可以塞进空档,和 mobile 不冲突。

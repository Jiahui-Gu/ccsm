# v0.3 design — daemon split

> **Status**: design spec, r12 final (0 P0 across 7 review angles, 7 review rounds completed; perMachine reconciliation locked at r12, 2026-05-01).
> **Scope**: refactor only — no added or removed features versus v0.2 (verified by feature-parity reviewer in r6 + r8). New behaviors (`daemon.shutdownForUpgrade` RPC, debugger inspect ports, install marker file) are all justified as P0 necessities of the daemon split per the feature-parity angle.
> **Source of truth**: the seven fragment files under `v0.3-fragments/` are the canonical spec. This file is the index/TOC. Companion exec summary: [`v0.3-daemon-split.md`](./v0.3-daemon-split.md). Plan: [`docs/superpowers/plans/2026-04-30-v0.3-daemon-split.md`](../plans/2026-04-30-v0.3-daemon-split.md). Plan-delta totals tracked in follow-up #941.

The v0.3 daemon split moves the long-running side of CCSM (PTY supervision, SDK calls, persistence) into a standalone daemon process, separate from the Electron app. The Electron client connects over a local IPC channel (length-prefixed JSON envelope in v0.3, Connect+Protobuf over HTTP/2 in v0.4). This document indexes the design spec; the implementation plan is `docs/superpowers/plans/2026-04-30-v0.3-daemon-split.md`.

## Canonical layout (Task #1043 dedup, 2026-05-01)

The spec was originally drafted as seven parallel fragments and then concatenated into this file. Both copies were maintained in parallel for several review rounds, which let drift creep in (frag-11 `perMachine` and frag-12 traceability rows were updated to r12 in PR #757 / #758 but the embedded copies here lagged behind).

To stop the drift permanently, **the fragment files are the single source of truth**:

- All section bodies (`§3.4.1`, `§3.5.1`, `§3.7`, `§6`, `§7`, `§8`, `§11`, `§12`) live in their owning `frag-*.md`.
- This file keeps only the intro, status, and the per-fragment index below.
- Any future spec edit lands in the fragment file and only the fragment file. Cross-references from code/tests use the existing `v0.3-design.md §X.Y` convention; readers follow the index here to the fragment file that owns `§X.Y`.

- **Code spec-ref convention**: source files reference sections as `v0.3-design.md §X.Y` (existing convention, preserved). Resolve via the section index above to the owning fragment.
